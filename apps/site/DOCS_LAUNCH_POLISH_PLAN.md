## Analytics recommendation

**Recommended: Plausible Analytics**, self-hosted on the existing homelab k8s /
DOKS cluster (the ecosystem is already self-hosting Market + Web Forge there).

Rationale:
- Privacy-respecting by design: no cookies, no cross-site tracking, no personal
  data, GDPR/CCPA-friendly with **no cookie-consent banner required** — consistent
  with the project's local-first / no-lock-in ethos.
- Lightweight script (~1 KB), single `<script defer data-domain>` tag — trivial to
  add to `BaseLayout.astro` and each Next.js root layout.
- Self-hostable (AGPL, Docker/Helm) → zero per-event SaaS cost; fits the existing
  k8s deployment pattern. Hosted Plausible Cloud is the alternative if you would
  rather not run it (~$9-19/mo at this traffic).

**Alternative: Umami** — also self-hostable, MIT-licensed, even lighter to run
(single container + Postgres, which DOKS already has). Pick Umami over Plausible if
you want MIT (vs AGPL) and the simplest possible self-host. Both are good; Plausible
has the more mature funnels/goals UI.

This is a **hosting + cost decision for the user**: choose self-host (free, more ops)
vs. cloud (small monthly fee, zero ops). Not implemented in this increment because
it needs that decision + a deployment target.

## PNG generation caveat

This environment has **no SVG→PNG rasterizer** available (`sharp` is not installed,
`rsvg-convert` / `cairosvg` absent). The increment therefore ships:
- the SVG OG-card source(s),
- a Node generation script (`scripts/generate-og.mjs`) that uses `sharp` when present,
- and, where PNGs could not be rasterized here, leaves the existing already-present
  PNGs in place (the site's default OG/favicon PNGs already exist and are correct).

To (re)generate or add new per-surface PNGs, run the documented command after
`pnpm add -D sharp` (one manual step, flagged below).

## Dark-mode recipe for the Next.js apps (market/profile/forge/auto-web)

The `@agentkitforge/ui` tokens already define `[data-theme="dark"]`, so each Next.js
app only needs to set that attribute on `<html>` before paint. Mirror what
`apps/site/src/layouts/BaseLayout.astro` does:

1. In the root `app/layout.tsx`, add a `<Script id="ak-theme" strategy="beforeInteractive">`
   (or a raw inline `<script dangerouslySetInnerHTML>` in `<head>`) that reads
   `localStorage('ak-theme')` else `prefers-color-scheme` and sets
   `document.documentElement.setAttribute('data-theme','dark')`. Inline + early to
   avoid FOUC.
2. Add a small client toggle button (the AppShell header is the natural home) that
   flips the attribute and persists `ak-theme`.
3. Any app that hardcodes light-only `--ak-*` overrides on `:root` must move those
   neutral overrides under an `html[data-theme="dark"]` block (the bug fixed on the
   site — `:root` overrides otherwise clobber the framework's dark tokens).
